# PROJECT AUDIT — Crypto Screener

## 1. Executive Summary
- One paragraph: what the app does, target user, end output.
- Programming language and framework (e.g., Python + Flask, Node.js + Express, etc.).
- Key libraries/dependencies used for data, math, charting, and UI.
- Overall architecture: data flow from exchange → detection → output.
- Current state in one sentence: “Detection works but has false positives and misses patterns.”

## 2. Project Structure
- Full file tree (files + one-line description for each).
- Main entry points and how to run the project.
- Dependency list with versions.
- Environment variables and secrets used.

## 3. Data Layer
- How OHLCV data is fetched (exchange/API, library, endpoints, websocket?).
- Which markets/symbols are scanned.
- Which timeframes are used.
- Data caching and storage (CSV, DB, in-memory, etc.).
- Rate limiting, error handling, and retries.
- Sample of the raw dataframe structure (columns, types, length).

## 4. Algorithm Catalog — Patterns, Levels, and Slopes
For EACH detection algorithm, paste:
- The exact function name and file path.
- The actual code (or exact pseudocode if the function is very long).
- All input parameters with their current default values.
- Step-by-step logic.
- What output it produces (coordinates, labels, scores, boolean flags, etc.).

Detection categories to cover (list every one that exists, and state "NOT IMPLEMENTED" if missing):
- **Chart patterns:** head & shoulders, inverse H&S, double top, double bottom, triple top/bottom, rounding bottom, ascending/descending/symmetrical triangle, wedge, flag, pennant, cup-and-handle, rectangle, channel, diamond, etc.
- **Support / resistance levels:** pivot highs/lows, fractals, volume profile, clustering (K-means, DBSCAN, mean-shift), rolling-window levels, psychological levels, etc.
- **Trendlines / slopes:** linear regression, Theil-Sen, RANSAC, local minima/maxima connection, Hough transform, polynomial fitting, moving slope, etc.
- **Smoothing / filtering / noise reduction:** SMA, EMA, WMA, Gaussian, Savitzky-Golay, Kalman, wavelet, ATR-based volatility filter, outlier removal, normalization, etc.
- **Signal scoring / ranking:** any confidence score, distance-to-level, pattern quality score, or ranking system.

## 5. Parameters & Configuration
- Create a table with every tunable parameter: name, file path, default value, allowed range, and what it controls.
- Highlight any "magic numbers" and explain why they were chosen.
- Note any parameters that are hard-coded and not configurable.

## 6. Output & User Interface
- How results are presented: console, web UI, JSON, CSV, alerts, charts, Telegram/Discord.
- Provide an example of the raw output for 2–3 symbols/timeframes (actual output if possible, or describe the format).
- Include screenshots if a UI exists, or describe the UI components.

## 7. Critical Performance Review (this is the most important section)
- Run the screener on a few representative symbols (e.g., BTC/USDT, ETH/USDT, SOL/USDT) across multiple timeframes.
- Show the raw signals/levels it produces.
- Evaluate the following specifically:
  - False positives: patterns/levels that appear but are not valid.
  - Missed patterns: obvious patterns that the algorithm did not detect.
  - Noise sensitivity: performance on small timeframes vs. higher timeframes.
  - Repainting / lookahead bias: does any code use future data by accident?
  - Lag: how many candles pass before a pattern is confirmed?
  - Robustness: how does it behave on choppy vs. trending markets?
- Identify the 3 weakest parts of the pipeline and explain why they are weak.

## 8. Improvement Recommendations
- List concrete algorithmic upgrades (e.g., use RANSAC for trendlines, DBSCAN for level clustering, multi-timeframe confirmation, machine-learning classifier for pattern validation, ATR-based dynamic thresholds, etc.).
- Code quality improvements (modularity, tests, type hints, logging).
- Performance optimizations (vectorization, caching, parallel scanning).
- Suggested validation / backtesting method to prove improvements reduce false positives.

## 9. Open Questions & User Decisions
- List anything unclear or that needs a user decision (e.g., target timeframe, risk tolerance, preferred UI).
