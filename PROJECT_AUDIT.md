## 1. Executive Summary

- **What the app does:** A real-time crypto chart **structure scanner**. The user uploads a screenshot of a chart (or selects a preset / draws pivots / picks a pattern type), the system extracts a *price structure* (pivot points, trend, geometry, pattern class), then scans the top Binance markets across 6 timeframes and returns the symbols whose structure is **visually/structurally most similar** to the reference. It also has independent scan modes for candlestick patterns, Fibonacci-level touches, and support/resistance levels. End output is a ranked list of `(symbol, timeframe, similarity %)` matches shown in a web UI with mini-charts.
- **Target user:** Crypto traders who want to find charts that "look like" a known pattern/setup without manually scanning hundreds of pairs.
- **End output:** Ranked match list (table + grid of sparkline charts) delivered over WebSocket, with detail modals (lightweight-charts candles, pivots, similarity score, detected patterns), plus a relevance feedback button.
- **Language / framework:** **Python 3.11** + **FastAPI** (ASGI, served by **uvicorn**). Frontend is a single static `index.html` (vanilla JS + Tailwind CDN + TradingView lightweight-charts + Chart.js). Real-time channel is a native FastAPI **WebSocket** at `/ws`.
- **Key libraries:**
  - Data: `aiohttp` (Binance/CoinGecko REST), `python-binance` (declared, not actively used for fetching — fetching is raw REST), `websockets` (declared).
  - Math/ML: `numpy`, `scipy` (signal/argrelextrema, spatial.distance), `scikit-learn` (declared; **no active model** found — see §4).
  - Image: `opencv-python-headless`, `pillow`.
  - Storage: `aiosqlite` (SQLite).
  - UI/charts: TradingView **lightweight-charts 4.1.0**, **Chart.js**, **Tailwind** (all via CDN).
- **Overall architecture (data flow):**
  ```
  Binance klines REST (data-api.binance.vision)  ─┐
  CoinGecko markets REST (top symbols, 24h chg)  ─┤
                                                  ▼
       BinanceScanner: in-memory candles per (symbol, timeframe)
                                                  ▼
       StructureExtractor: normalize → smooth → pivots(zigzag/multiscale)
                           → trend/compression/geometry → classify pattern
                           (full window + sliding windows 30/50/70)
                                                  ▼
   ┌─ similarity mode: SimilarityMatcher.find_matches(reference, structures)
   ├─ type_scan mode:  filter by structure_type / detected_patterns
   ├─ candle_scan:     CandlePatternDetector.find_all_positions
   ├─ fibo_scan:       FibonacciAnalyzer
   └─ level_scan:      LevelDetector (S/R + trendlines)
                                                  ▼
       FastAPI routes + WebSocket broadcast → static/index.html UI
                                                  ▼
       SQLite (uploaded_structures, matches, feedback, scan_sessions)
  ```
- **Current state (one sentence):** Detection works and produces ranked results, but it relies on many hand-tuned magic-number thresholds operating on a **resampled 100-point normalized line**, has meaningful false-positive and lookahead/repaint risk, ships a **dormant synthetic-data generator** (defined but not currently wired into the fetch path), and has **no backtesting or validation harness**.

## 3. Data Layer

### 3.1 How OHLCV is fetched

- **Exchange/API:** Binance **public data** mirror, not the main API:
  - Klines: `https://data-api.binance.vision/api/v3/klines`
  - 24h tickers: `https://data-api.binance.vision/api/v3/ticker/24hr`
- **Library:** raw `aiohttp` GET requests (NOT `python-binance`, despite it being a dependency).
- **Symbol list source:** **CoinGecko** `https://api.coingecko.com/api/v3/coins/markets` (`vs_currency=usd`, `order=market_cap_desc`, `per_page=100`) is used both for the top-symbols list and for 24h price-change badges. There is a separate CoinGecko-driven `/api/market-movers` endpoint cached for **300 s**.
- **WebSocket to exchange?** **NO.** Despite `replit.md` describing "Binance WebSocket API for real-time monitoring", the actual implementation is **REST polling** (see §3.5). The only WebSocket in the system is the **server↔browser** channel (`/ws`).

### 3.2 Markets / symbols scanned

- Top **50** symbols (`num_symbols = min(num_symbols, 50)`, default 50), resolved via `fetch_top_symbols()` (CoinGecko market-cap order, filtered to valid Binance USDT symbols via `_is_valid_binance_symbol`). A hardcoded `_get_default_symbols()` fallback list exists if fetching fails.

### 3.3 Timeframes

- `TIMEFRAMES = {"1m":1, "5m":5, "15m":15, "1h":60, "4h":240, "1d":1440}` (value = minutes). All 6 are scanned for every symbol.
- **Candles per series:** `limit=100` (default everywhere).
- **Sliding sub-windows:** `WINDOW_SIZES = [30, 50, 70]`. For each `(symbol, timeframe)` the extractor is also run on the **last 30/50/70 candles**, registered as pseudo-timeframes named `"{tf}_w{N}"` (e.g. `1h_w30`). This multiplies the structure count ~4× per series.

### 3.4 Caching & storage

- **In-memory:** `scanner.symbol_data[symbol]` holds `candles[timeframe] -> List[CandleData]` and `structures[timeframe] -> StructureFeatures`. This is the live working set used for matching.
- **SQLite (`data/structures.db`)** via `aiosqlite`. Schema (exact, from `database.py`):
  - `uploaded_structures(id, name, structure_type, feature_vector TEXT, normalized_line TEXT, pivot_sequence TEXT, relative_distances TEXT, trend_direction REAL, volatility REAL, compression_ratio REAL, created_at)`
  - `matches(id, structure_id, symbol, timeframe, similarity_score, structure_type, is_mirrored, normalized_line TEXT, matched_at)`
  - `feedback(id, match_id, is_relevant, created_at)`
  - `scan_sessions(id, structure_id, started_at, stopped_at, threshold, total_matches)` (created but lightly used)
- **Uploaded images:** saved to `uploads/{md5[:8]}_{filename}`.
- **CSV:** **NOT USED.**
- Note: only **uploaded** reference structures and their matches persist; live market structures are **in-memory only** and recomputed.

### 3.5 Rate limiting, error handling, retries

- **Concurrency throttle:** `_update_all_candles` fetches in batches per timeframe with `await asyncio.sleep(0.15)` between batches; uses a shared `aiohttp.ClientSession`.
- **Poll loop:** `_poll_loop` sleeps **60 s** between full refreshes; on exception sleeps **10 s** and retries (loop-level try/except).
- **Per-fetch error handling:** `_fetch_binance_klines` wraps in try/except, prints the error, and returns `[]` on failure.
- **Retries:** No explicit exponential backoff / retry counter. A failed fetch simply yields empty until next poll.
- **⚠ Dormant simulation generator:** `_generate_realistic_candles(symbol, timeframe, limit=100)` fabricates synthetic OHLCV (seeded by symbol, random "pattern_seed" branches). **It is defined but currently NOT CALLED anywhere** in the fetch path (verified by grep — only the definition exists). So the runtime does **not** silently fall back to fake data today; if real data is unavailable, the series is simply empty and that `(symbol, timeframe)` produces no structure. The risk is latent: re-wiring this function into the fetch path (as the old design implied) would let matches be produced against fabricated data.

## 4. Algorithm Catalog — Patterns, Levels, and Slopes

> **Critical context for accuracy review:** Almost all *structure/pattern* detection runs on a **normalized, resampled 100-point line** (`normalize_line`) derived from closes, then **smoothed** (`_adaptive_smoothing`, Savitzky-Golay). Pivots are detected on this transformed line, not on raw OHLC. This is the single biggest driver of both robustness and false positives.

### 4.0 Preprocessing — normalization, smoothing, resampling

**File:** `structure_extractor.py`

- `normalize_line(line)`:
  - Resamples to `resample_points = 100` via `np.interp`.
  - Min-max normalizes to [0,1]: `(line - min) / (max - min)`; if range==0 returns zeros.
- `_adaptive_smoothing(line)`:
  - **Savitzky-Golay** (`scipy.signal.savgol_filter`) applied in up to **two passes**.
  - Window 1 derived from length; forced odd; min 3.
  - Second pass window (`window2`) gated by a `noise_ratio` estimate; forced odd; min 3.
  - **Output:** denoised 100-pt line used for pivot detection.

#### Smoothing / filtering / noise reduction inventory
- **SMA / EMA / WMA:** NOT IMPLEMENTED as named filters (no moving-average smoothing of the price series).
- **Gaussian:** NOT IMPLEMENTED.
- **Savitzky-Golay:** **IMPLEMENTED** (`_adaptive_smoothing`, adaptive window).
- **Kalman / wavelet:** NOT IMPLEMENTED.
- **ATR-based volatility filter:** **IMPLEMENTED implicitly** inside zigzag (`atr`, `atr_mult` derived from per-bar noise — see 4.1).
- **Outlier removal:** NOT IMPLEMENTED as a dedicated step (normalization + smoothing only).
- **Normalization:** **IMPLEMENTED** (min-max to [0,1] on resampled 100-pt line).

### 4.1 Pivot detection (the backbone of everything)

**Function:** `StructureExtractor.detect_pivots(line, order=None)` → `List[PivotPoint]`
**Pivot dataclass:** `index, value, is_high, relative_position, prominence, confidence`.

**Logic (step by step):**
1. Compute zigzag pivots (`_zigzag_pivots`) and multi-scale pivots (`_multi_scale_pivots`) on the smoothed line.
2. **If `len(zigzag_pivots) >= 3`:** start from the zigzag set and **append** any multi-scale pivot whose `prominence > prominence_threshold` **and** that is not within `min_dist = max(3, len/25)` of an existing pivot, where
   `prominence_threshold = max(0.08, min(0.25, 0.12 + diff_noise * 2.0))` (dynamic, noise-scaled). Then sort by index.
3. **Else (zigzag insufficient):** use the multi-scale set if it has ≥3 pivots, otherwise the fallback set (`_fallback_pivots`).
4. `_calculate_prominence` annotates prominence; `_filter_redundant_pivots` prunes; `_calculate_confidence` annotates confidence.
5. If pivot count > `num_pivots` (10), `_select_important_pivots` trims to the most important.

**`_zigzag_pivots(line)` — key params/logic:**
- `noise_ratio` computed from per-bar differences.
- `atr` = mean absolute bar-to-bar change.
- `atr_mult = min(3.5, max(1.5, 1.5 + noise_ratio * 0.8))` → **adaptive 1.5×–3.5× ATR**.
- `min_swing = max(atr * atr_mult, 0.02)` → minimum reversal amplitude to register a pivot.
- `min_spacing` enforces minimum index gap between pivots.
- Standard zigzag state machine: track running high/low; flip direction when `last_high_val - line[i] >= min_swing` (down) or `line[i] - last_low_val >= min_swing` (up).

**`_multi_scale_pivots(smoothed, original, adaptive_order=3)`:**
- Uses `scipy.signal.argrelextrema` at the adaptive order to find local max/min, dedup within ±2 indices.

**`_adaptive_order(line)`:** picks the argrelextrema comparison order from line characteristics.

**`_calculate_confidence(pivots, line, ...)`:** per-pivot confidence ∈ [0,1]:
- `prominence_score = min(1.0, p.prominence * 2.0)`
- multi-scale `confirmation_count` (pivots confirmed within ±3 idx)
- amplitude term
- weighted: `prominence_score * 0.35 + ... ` (35% prominence + confirmation + amplitude).

**`_select_important_pivots`:** scores each pivot
`score = (prominence*0.5 + neighbor_diff*0.3 + 0.2*position_weight) * position_weight`
with `position_weight` boosted (1.5 / 1.3) for pivots near the **end** of the window (recency bias).

**Output:** ordered list of pivots with index/value/is_high/prominence/confidence.

> **Support/Resistance — pivot highs/lows / fractals:** pivot detection here is zigzag + argrelextrema fractals. **K-means / DBSCAN / mean-shift / volume-profile clustering:** NOT IMPLEMENTED in the structure module (see §4.6 for the separate `LevelDetector`, which uses simple price-bucket clustering, not sklearn).

### 4.2 Trend, volatility, compression, quality

**File:** `structure_extractor.py`
- `calculate_trend(line)` → slope of a linear fit (normalized), used as `trend_direction`.
- `calculate_volatility(line)` → dispersion measure of the normalized line.
- `calculate_compression(line, pivots)` → ratio of late-window volatility to early-window volatility (`vol_end / vol_first`); <1 means contracting (consolidation).
- `calculate_quality_score(line, pivots)`:
  - alternation count of highs/lows,
  - `avg_prominence` → `prominence_score = min(1.0, avg_prominence * 5)`,
  - combined: `... + prominence_score * 0.25 + ...` → overall structure quality ∈ [0,1].
  - `self.min_quality = 0.15` is the floor used in gating.

### 4.3 Chart-pattern detectors (the 20-class catalog)

All live in `structure_extractor.py` and operate on the normalized/smoothed 100-pt line + pivots. Each returns `(is_detected, confidence, ...)` and most also return an **active/freshness** flag. `classify_structure(trend, compression, pivots, line)` orchestrates them and outputs the **primary** `StructureType` plus a **`detected_patterns` dict** (multi-label).

**`StructureType` enum (20 values):** `compression, accumulation, triangle, range, retest, trend_up, trend_down, impulse_up, impulse_down, breakout, squeeze_up, squeeze_down, double_top, double_bottom, head_shoulders, inv_head_shoulders, bull_flag, bear_flag, rising_wedge, falling_wedge` (+ `unknown`).

#### Double Top — `_detect_double_top(pivots, line)`
- Iterate high-pivot pairs; `diff = |top1 - top2| / price_range`.
- Accept only if `diff < 0.03` (**3% tolerance**, tightened from a former 10%).
- Require a valley dip between tops; `conf = (1 - diff/0.03)*0.5 + min(1, dip_depth/0.3)*0.5`.
- Reject if `best_conf <= 0.4`.
- **Freshness:** flagged inactive if price already broke the neckline.
- **Output:** `(is_double_top, confidence, is_active)`.

#### Double Bottom — `_detect_double_bottom(pivots, line)`
- Symmetric to double top: `diff < 0.03`, `conf = (1 - diff/0.03)*0.5 + min(1, peak_height/0.3)*0.5`, reject `<= 0.4`.

#### Head & Shoulders — `_detect_head_shoulders(pivots, line)`
- For triplets of highs (L, head, R):
  - `shoulder_diff = |left - right| / price_range`; `head_prominence = (head - max(left,right))/price_range`.
  - Require `shoulder_diff < 0.18` **and** `head_prominence > 0.06`.
  - `conf = (1 - shoulder_diff/0.18)*0.3 + min(1, head_prominence/0.15)*0.4 + neckline_bonus + 0.1`
    where `neckline_bonus = 0.2` if ≥2 neckline lows.
  - Proportion penalties: `shoulder_head_ratio = shoulder_avg/head`; if `<0.4` or `>0.95` → `conf *= 0.6`. Time-symmetry penalty `conf *= 0.7` if asymmetric.
  - Reject `best_conf <= 0.4`.

#### Inverse H&S — `_detect_inv_head_shoulders(pivots, line)`
- Mirror of H&S on low pivots; `head_prominence = (min(left,right) - head)/price_range`; same `0.18`/`0.06` gates; depth-ratio penalty if `depth_ratio < 0.15` or `> 0.85`.

#### Flag (bull/bear) — `_detect_flag(line, pivots)`
- Detects a **pole** then a counter-trend **flag** channel.
- Bull: `pole_move > 0 and flag_trend <= 0`; Bear: `pole_move < 0 and flag_trend >= 0`.
- Proportionality: pole must dominate the flag (replit.md: pole ≥1.5× flag).
- Reject `best_conf <= 0.5`.
- **Freshness:** inactive if tail mean breaks `flag_high + 0.03*range` (bull) / below `flag_low - 0.03*range` (bear).
- Output `(is_flag, direction, conf, is_active)`.

#### Wedge (rising/falling) — `_detect_wedge(pivots, line)`
- Fit upper & lower bounds; `convergence = 1 - spread_end/spread_start`; require `convergence >= 0.15`; `fit_score >= 0.7`.
- `conf = convergence*0.4 + fit_score*0.4 + 0.2`.
- Counts breakouts for freshness.

#### Triangle — `_detect_triangle(pivots, line, compression)`
- Fit highs line + lows line (needs ≥3 of each).
- `convergence_ratio = 1 - spread_end/spread_start`; require `>= 0.15` **and** `<= 0.95` (upper cap to avoid degenerate full-convergence).
- Normalized convergence rate cap: `norm_conv_rate = convergence_ratio/(pattern_span/n)` constrained (`< 3.0`).
- `h_fit/l_fit` from residuals; `fit_score = (h_fit+l_fit)/2`; require `fit_score >= 0.75`.
- `conf = convergence_ratio*0.35 + fit_score*0.35 + pivot_count_score*0.2 + max(0,compression)*0.1`.

#### Squeeze (up/down) — `_detect_squeeze(pivots, line)`
- Detects volatility contraction with directional bias → `squeeze_up`/`squeeze_down`.

#### Impulse (up/down) — `_detect_impulse(line)`
- Strong directional move detection; scans for best confidence over the move; returns `(is_impulse, direction, conf)`.

#### Trend up/down, Range, Compression, Accumulation, Breakout, Retest
- Derived in `classify_structure` from `trend_direction`, `compression`, pivot structure and HH/HL vs LH/LL ratios:
  - Uptrend branch: `overall_move > 0.15 and (hh_ratio >= 0.5 or hl_ratio >= 0.5)` → `trend_up`.
  - Downtrend branch: `overall_move < -0.15 and (lh_ratio >= 0.5 or ll_ratio >= 0.5)` → `trend_down`.
  - Range/accumulation: low high-range & low low-range (`< 0.15` each).
  - **breakout** is assigned from breakout-strength logic.
  - **`retest`:** the value exists in the `StructureType` enum but is **NEVER emitted** by `classify_structure` (no detector assigns it). Effectively **NOT IMPLEMENTED** as an output class (it can still be selected as a *reference* preset/type-scan filter, but no market structure is ever classified as `retest`).

#### Convergence rate — `calculate_convergence_rate(pivots, line)`
- Needs ≥2 highs and ≥2 lows. `spread_start = |highs[0] - lows[0]|`, `spread_end = |highs[-1] - lows[-1]|`; `rate = (spread_start - spread_end)/spread_start`, clipped to [-1, 1]. (Positive = converging.)
- **Note:** the `line[:int(n*0.75)]` vs `line[int(n*0.75):]` split is used by a **different** function, `calculate_breakout_strength`, not by convergence rate.

### 4.4 Multi-label classification & freshness

- `classify_structure(...)` returns `(StructureType, pattern_confidence, detected_patterns, is_active, pattern_freshness)`.
- `detected_patterns` is a `{type: confidence}` dict — **all** patterns that passed their gate, not just the primary. This is used to relax type penalties in matching and in `type_scan`.
- **Freshness/active flag:** patterns are marked inactive if price already broke their boundary (neckline / triangle edge / flag channel). Inactive structures are filtered out in `find_matches` (`is_pattern_active` check) and contribute a score penalty if `pattern_freshness < 0.5`.

### 4.5 Feature vector & extract_features

- `extract_features(price_line)` → `StructureFeatures` dataclass containing: `normalized_line`, `pivot_points`, `pivot_sequence`, `relative_distances`, `pivot_slopes`, `pivot_angles`, `feature_vector`, `structure_type`, `detected_patterns`, plus scalar scores: `quality_score, pattern_confidence, symmetry_score, convergence_rate, breakout_strength, avg_pivot_confidence, trend_consistency, pattern_freshness (default 1.0), volume_confirmation (default 0.5)`.
- `feature_vector` blends the normalized line (scaled ×0.35) with scalar features (`scalar_features * 0.15`) for vector comparison.
- **Volume confirmation** computed from pivot-volume, volume trend, and recent volume strength (0–1).

### 4.6 Support / Resistance levels & trendlines — `level_detector.py`

**Class:** `LevelDetector(deviation_pct=0.15, min_touches=3)`
**Main:** `detect_levels(highs, lows, closes, ..., min_touches=None)`
1. `_find_pivots(data, is_high)` — local extrema via rolling window (`window = data[i-order : i+order+1]`, `data[i]==max/min`). (Fractal pivots, NOT sklearn.)
2. `_find_lines(...)` for resistance (high pivots) and support (low pivots):
   - `tolerance = price_range * (deviation_pct/100.0)`
   - `breakout_threshold = price_range * 0.003`
   - Group pivots into candidate lines; require `len(touches) >= min_touches`.
   - Score: spacing score + `(1 - avg_dev_pct/deviation_pct)*30` + touch count.
3. `_count_touches` / `_count_breakouts` quantify validity; `_deduplicate` merges same-type lines whose **relative midpoint distance** `|val_lv - val_k| / (|k.intercept| + 1e-10) < 0.005` **and** slope diff `< 0.01` (lines are evaluated at the window midpoint `mid = n//2` using `slope*mid + intercept`).

- **Trendlines / slopes available:** local-extrema connection + linear fitting of touch points (effectively least-squares line through clustered pivots), with a slope term enabling **diagonal** trendlines (not just horizontal).
- **NOT IMPLEMENTED:** RANSAC, Theil-Sen, Hough transform, polynomial fitting, K-means/DBSCAN/mean-shift clustering, volume-profile levels, psychological round-number levels.

### 4.7 Fibonacci — `fibonacci_analyzer.py`

**Class:** `FibonacciAnalyzer(touch_tolerance=0.003, min_swing_pct=0.02)`
- Retracement levels: `0.0, 0.236, 0.382, 0.5, 0.618, 0.786, 1.0`; extensions: `127.2%, 161.8%, 261.8%` (projected beyond the swing — higher for uptrends, lower for downtrends).
- `find_swing_points` / `find_major_swings` locate swing high/low.
- `detect_touches(...)`: a candle touches a level if `deviation <= touch_tolerance (0.003)`; bounce detected via `expected_move = level_price * touch_tolerance * 2`.
- `score_quality(touches, total_candles)`:
  `touch_score*0.25 + level_score*0.20 + bounce_ratio*0.20 + precision_score*0.15 + key_score*...`
  where `precision_score = max(0, 1 - avg_deviation/0.3)`, `level_score = min(1, unique_levels/3)`, `key_score = key_levels_hit/3` (key = 38.2/50/61.8).

### 4.8 Candlestick patterns — `candle_patterns.py`

**Class:** `CandlePatternDetector` — 19 patterns (enum `CandlePatternType`):
`doji, hammer, inverted_hammer, shooting_star, spinning_top, marubozu_bull, marubozu_bear, engulfing_bull, engulfing_bear, morning_star, evening_star, three_white_soldiers, three_black_crows, harami_bull, harami_bear, piercing_line, dark_cloud, tweezer_top, tweezer_bottom`.

- Geometry helpers: `_body, _upper_shadow, _lower_shadow, _is_bullish, _is_bearish, _range`.
- Example thresholds: doji `body_ratio < 0.1`; hammer `body_ratio<0.1`, `upper<=body*0.5`, `body_ratio<0.4`; marubozu `body_ratio>0.85`, shadows `/rng < 0.05`; spinning top `0.1<=body_ratio<=0.35`, both shadows `/rng > 0.2`.
- **Trend-context filter:** `_detect_preceding_trend(candles, idx, lookback=5)` → bullish reversal patterns only valid after a downtrend, bearish only after uptrend (`up_count/down_count > total*0.6`).
- **Volume filter:** in `find_all_positions(..., max_age=10)`: `vol_ratio = candle.volume/avg_vol`; patterns with `vol_conf < 0.6` are dropped **except** `doji`/`spinning_top`.
- **Recency:** only scans `start_idx = max(2, len(candles) - max_age)` → last **10** candles by default.
- **Output:** list of dicts `{index, time, pattern, ...}` for UI overlay; market scan mode (`candle_scan`) broadcasts matches per symbol.

### 4.9 Image processing — `image_processor.py`

**Class:** `ImageProcessor`
- `detect_chart_area(img)`: estimates UI margins via Canny edge density (`cv2.Canny(strip,50,150)`), crops top 15% / right 15% / bottom 15% if `density > 0.05`.
- `extract_candlestick_prices(img)`: HSV color masks for green candles `[35,80,80]–[85,255,255]` and red `[0/170,80,80]–[10/180,255,255]`; morphological close; per-column weighted price (`green_weight 0.7`, `red_weight 0.3`); interpolates gaps; rejects if candle pixels `< 0.005 * area`.
- `extract_price_line_by_color(img)`: tries several line-color ranges (for line charts) and keeps the best mask.
- `extract_price_line(bytes)` (called by `/api/upload`): orchestrates the above; returns a 1-D price array; upload rejected if `< 20` points extracted.

### 4.10 Similarity scoring — `similarity_matcher.py` (the ranking engine)

**Class:** `SimilarityMatcher`
**Component weights (sum to 1.0):**
`line_weight 0.25, pivot_weight 0.20, distance_weight 0.10, shape_weight 0.25, slope_weight 0.10, geometry_weight 0.10`.

**`compare_lines(line1, line2)`** (sub-weights):
`0.20*correlation + 0.15*mse_similarity + 0.15*max(0,cosine) + 0.30*dtw_sim + 0.20*segment_sim`.
- `_dtw_distance(s1, s2, window=...)` — banded DTW (Sakoe-Chiba). In `compare_lines` the inputs are **downsampled** (`line[::4]` if len≥20 else `line[::2]`) and the band is **dynamic**: `window = max(3, len(down1)//5)`. The `window=10` is only the function's default, not what `compare_lines` passes. Also: `correlation` is **Pearson** (`pearsonr`, rescaled `(r+1)/2`), `mse_similarity = 1/(1+mse*10)`, `cosine` via `scipy.spatial.distance.cosine`, `segment_sim` = mean Pearson over 4 segments.

**`compare_pivots`** → `0.40*mse_sim + 0.35*direction_match + 0.25*magnitude_sim` (returns 0.3 if missing).
**`compare_distances`** → relative-distance similarity (0.3 if missing).
**`compare_slopes`** → `0.5*slope_sim + 0.5*angle_sim`.
**`compare_geometry`** → `0.25*(symmetry + convergence + breakout + trend)`.
**`compare_shape`** → `0.20*trend_sim + 0.10*vol_ratio + 0.20*comp_sim + 0.50*type_sim`, where `type_sim`:
- same type → `1.0`
- same category (`_same_category`) → `0.60`
- opposites (`_are_opposites`) → `0.15`
- any other mismatch → `0.35`
- then `type_sim = min(1.0, type_sim + overlap)` where `overlap = _pattern_overlap_bonus = min(0.3, best_overlap*0.4)` from `detected_patterns` overlap (this **relaxes** the type penalty).

**Mirror handling — `calculate_similarity(reference, candidate, check_mirror=True)`:**
- Computes normal score and a mirrored-candidate score; mirror score multiplied by:
  - `0.85` if opposites, `0.95` if both consolidation types, else `0.90`.
- Returns the higher of normal vs mirror (×100), plus `is_mirrored` flag.

**Gates — `_apply_gates(score, ref, cand)`:**
- `type_gate` (`_calculate_type_gate`) multiplies score.
- **Confidence multiplier** (`_confidence_multiplier`): vs `MIN_CONFIDENCE_THRESHOLDS[type]` (e.g. IMPULSE 0.55, H&S 0.50, flags 0.50, DOUBLE_TOP/BOTTOM 0.45, TRIANGLE 0.40, COMPRESSION 0.35, ACCUMULATION 0.30, default 0.30):
  - `conf < min*0.5` → ×0.6; `conf < min` → ×0.8; else ×1.0. Uses `min(ref_mult, cand_mult)`.
- Quality gate: `avg_quality < 0.3` → ×0.8; `< 0.4` → ×0.9.
- Pivot-confidence gate: `>0.6` → ×1.05; `<0.3` → ×0.92.
- Freshness gate: `cand_freshness < 0.5` → ×`(0.6 + freshness*0.8)`.
- Volume gate: `vol > 0.7` → ×`(1 + (vol-0.7)*0.3)`; `vol < 0.3` → ×0.85.
- Final `min(1.0, score)`.

**`find_matches(reference, candidates, threshold=50.0)`:**
- Skips candidates with `quality_score < 0.2` or `is_pattern_active == False`.
- Keeps matches with `score >= threshold`.
- `_deduplicate_matches`: best per `(symbol, base_tf)`; near-duplicates within 5.0 score collapsed.
- `_apply_mtf_bonus`: same symbol on ≥3 base timeframes → ×1.08; ≥2 → ×1.04 (capped 99.99).
- Sort descending by score.

**Signal scoring / ranking summary:** the final 0–100 similarity is the gated, MTF-boosted weighted blend above. Confidence (per pattern) and pivot confidence are separate [0,1] scores. **No ML classifier / no learned ranking** — `scikit-learn` is unused despite being a dependency.

### 4.11 MTF logic in `main.py`
- `type_scan` path recomputes its own MTF bonus (×1.08 for ≥3 TF, ×1.04 for ≥2 TF) and applies a volume adjustment to confidence (`vol>0.7` boost, `vol<0.3` ×0.85) before broadcasting.

## 7. Critical Performance Review (most important section)

> Because the live scanner is in-memory and the audit is static, the assessment below is derived from reading the exact algorithms and tracing how representative inputs (BTC/ETH/SOL across 1m→1d) flow through them. Where a behavior is a definite code property (not market-dependent) it is stated as such.

### 7.1 False positives (patterns/levels that appear but are not valid)
- **Resampling to 100 points distorts proportions.** Every series — whether 30 or 100 candles — is stretched/compressed to 100 points and min-max normalized. A shallow wiggle on a 1m chart and a major multi-week structure on 1d become **geometrically identical** after normalization, so the similarity engine happily matches them. This is a structural false-positive generator.
- **Sliding windows inflate match count & duplicates.** Each series spawns 4 structures (full + 30/50/70). Dedup is per `(symbol, base_tf)` with a 5-point collapse window, but the same underlying move can still surface as multiple "patterns".
- **Pattern-overlap bonus + relaxed type penalty.** `type_sim` only drops to 0.15 for true *opposites* (0.35 for any other type mismatch, 0.60 for same-category), and `_pattern_overlap_bonus` (up to +0.3) can push it back up — so **structurally different charts still clear the 50% threshold**. The matcher is tuned toward recall, not precision.
- **Mirror matching at ×0.85–0.95.** A bearish chart can match a bullish reference at up to 95% of its mirror score → "matches" that are visually inverted.
- **Triangle/wedge on few pivots.** With only 3 fitted points per side, a near-random zigzag can produce `convergence ≥ 0.15` and `fit ≥ 0.75` and be labeled a triangle.
- **Dormant synthetic-data generator (latent risk).** `_generate_realistic_candles` can fabricate charts, but it is **not currently called**, so today an unavailable symbol simply yields no structure rather than fake matches. The risk is latent: if this function is ever re-wired into the fetch path, it would create **false positives by construction**.

### 7.2 Missed patterns (obvious patterns not detected)
- **No triple top/bottom, cup-and-handle, pennant, channel, diamond, rounding bottom** detectors at all.
- **Triangle sub-types collapsed:** ascending vs descending vs symmetrical are one `triangle` label — a user searching specifically for an ascending triangle cannot.
- **Strict fits on noisy TFs:** `fit ≥ 0.75` (triangle) / `0.7` (wedge) on a smoothed but still jagged 1m line will reject many valid-but-rough patterns.
- **Recency bias in pivot selection** (`position_weight` 1.3–1.5 near the window end) can drop a clean early-window pattern.
- **Candle scan only inspects the last 10 candles** (`max_age=10`) — older valid signals are invisible.

### 7.3 Noise sensitivity (low vs high timeframe)
- Adaptive zigzag (`atr_mult` 1.5–3.5 by noise) and Savitzky-Golay partly compensate, but **all downstream gates are fixed constants on the normalized line**, so 1m (noisy) tends to **over-trigger trend/impulse/breakout** while higher TFs are cleaner. There is no per-timeframe parameter set.

### 7.4 Repainting / lookahead bias
- **Whole-window detection = inherent repaint.** Patterns are detected on the **entire current 100-pt window**, including the most recent candle, and pivots near the end get **boosted weight**. As new candles arrive and the window re-normalizes, previously reported pivots/patterns can shift or disappear. This is classic **repainting**; nothing here is computed in a strictly causal, bar-by-bar confirmed manner.
- **Freshness flag is computed from the same window** (did price already break the boundary?), which uses the latest bar — it mitigates "already-broken" stale patterns but does not eliminate repaint of the pattern shape itself.
- No explicit *future* leakage (no code indexes `i+future`), but the **window re-fit on every poll** is functionally lookahead-sensitive from a backtest standpoint.

### 7.5 Lag (candles to confirmation)
- A pattern can appear **immediately** (no N-bar confirmation rule) because detection is on the full window — i.e. **near-zero lag but high repaint**. The freshness/active check is the only "confirmation," and it can flip as the next candle prints. There is no "wait for close beyond neckline + retest" confirmation gate.

### 7.6 Robustness (choppy vs trending)
- **Trending:** trend/impulse/breakout branches fire reliably (`±0.15` move + HH/HL ratios).
- **Choppy:** zigzag adapts swing size, but range/accumulation vs triangle/squeeze classification becomes unstable; small thresholds (`0.15`) on a flat normalized line are easily tripped, producing label churn between `range`, `compression`, `accumulation`, `triangle` poll-to-poll.

### 7.7 The 3 weakest parts of the pipeline
1. **Normalization + resampling to 100 points (`normalize_line`).** It destroys absolute scale, time, and volatility context, making cross-timeframe/cross-symbol matches meaningless in real-trading terms and feeding every downstream false positive. **Weak because** the whole matching premise rests on a representation that discards the information a trader actually uses.
2. **Whole-window, non-causal pattern detection (repaint).** Patterns re-fit on every poll over the full window with recency-weighted pivots → results are unstable and not reproducible in a live/backtest setting. **Weak because** signals can't be trusted to persist or to have existed at the claimed time.
3. **Hand-tuned fixed thresholds + recall-biased similarity gating (no validation).** Dozens of magic numbers (`0.03`, `0.18`, `0.15–0.95`, type_sim 1.0/0.60/0.35/0.15, overlap +0.3, mirror 0.85–0.95) with **no backtest/labelled dataset** to justify them. **Weak because** there is no objective feedback loop proving any threshold reduces false positives; the stored user feedback is never used.

## 8. Improvement Recommendations

### 8.1 Algorithmic upgrades
- **Trendlines:** replace least-squares-through-clustered-pivots with **RANSAC** or **Theil–Sen** line fitting for outlier-robust support/resistance/trendlines; optionally **Hough transform** on the pivot scatter for diagonal lines.
- **Level clustering:** replace fixed price-bucket grouping with **DBSCAN** or **mean-shift** on pivot prices (bandwidth from ATR); add **volume-profile** (POC/value-area) and **psychological round-number** levels.
- **Causal detection:** detect patterns on a **rolling, closed-bar** basis and require a **confirmation rule** (e.g. close beyond neckline + optional retest) before emitting. Tag each signal with the **bar index it was confirmed on** to eliminate repaint and enable honest backtesting.
- **Per-(symbol,timeframe) adaptive thresholds:** make every gate a function of **ATR / realized volatility**, not constants on the normalized line. Keep absolute price/time context instead of resampling to 100; if a fixed length is needed, store scale & duration as features and penalize cross-scale matches.
- **Pattern validation classifier:** the existing geometric detectors can generate candidate windows; train a **lightweight ML classifier** (start with the already-present `scikit-learn`) on user-labeled "relevant/irrelevant" feedback to filter false positives — this finally *uses* the feedback table.
- **Distinguish triangle sub-types** (ascending/descending/symmetrical) and **add missing patterns** (triple top/bottom, cup-and-handle, pennant, channel, rounding bottom).
- **Tighten similarity for precision:** raise type-mismatch penalty, cap the overlap bonus, and make the mirror penalty configurable; expose a precision/recall slider.
- **Remove or hard-flag the simulation fallback:** never emit matches against synthetic data without an unmissable UI warning, or disable scanning entirely when `data_available == False`.

### 8.2 Code-quality improvements
- Add **type hints consistently** and **docstrings** on detector functions; extract all magic numbers into a single **`config.py`/dataclass** (or YAML) so they are discoverable and tunable.
- Split `structure_extractor.py` (1626 lines) and `main.py` (1449 lines) into smaller modules (one detector per file; routes vs orchestration vs scan-callbacks).
- Add **structured logging** (replace `print`) with levels and per-symbol context.
- Add **unit tests** with synthetic-but-labeled charts for each detector (golden cases + known negatives), and **regression tests** on the similarity weights.

### 8.3 Performance optimizations
- **Vectorize** pivot/feature computation; precompute reference feature vector once per scan.
- **Cache** structures by candle hash (a `_candle_hash` already exists — extend it to skip recompute when unchanged).
- **Parallelize** scanning (asyncio gather / process pool) and consider an actual **Binance WebSocket** stream to replace 60 s REST polling (matches the README's stated design and cuts latency).

### 8.4 Validation / backtesting to prove fewer false positives
- Build a **labeled dataset** from the existing feedback table + manually tagged historical windows.
- Implement an **event-study backtest:** for each emitted signal, measure forward return / hit-rate at a fixed horizon, and compute **precision, recall, F1** per pattern and per timeframe.
- Compare **before/after** any threshold change on the same frozen dataset; only ship changes that improve precision without large recall loss. Track **repaint rate** (how often a reported signal disappears on the next bar) as a first-class metric.

## 9. Open Questions & User Decisions

1. **Primary goal:** Is the core value the *screenshot→similar-charts* matcher, or the *classical pattern/level/Fib scanners*? They have different accuracy roadmaps; which should be optimized first?
2. **Target timeframe(s):** Should the system focus on a few TFs (e.g. 1h/4h/1d for swing trading) so thresholds can be tuned per-TF, instead of one-size-fits-all across 1m→1d?
3. **Precision vs recall:** Do you prefer **fewer, higher-quality** matches (raise gates, drop mirror/overlap bonuses) or **more candidates** to eyeball? This sets the entire tuning direction.
4. **Repaint tolerance:** Is it acceptable for a reported pattern to change as new candles arrive, or do you require **confirmed, non-repainting** signals (with some lag)?
5. **Simulation fallback:** Should the app **refuse to scan** when real Binance data is unavailable, rather than fall back to synthetic candles?
6. **Cross-scale matching:** Should matches be allowed across different timeframes/durations at all, or should scale/time be preserved and enforced?
7. **Use of feedback:** Do you want the "Релевантно/Нерелевантно" feedback to actually **train a filter** (requires storing features per match and a retraining step)?
8. **Symbol universe:** Keep top-50 by market cap (CoinGecko), or scan a custom/larger Binance USDT universe (affects rate limits)?
9. **Output channels:** Do you need alerts (Telegram/Discord) or CSV/JSON export in addition to the web UI?
10. **Missing patterns:** Which of the not-implemented patterns (triple top/bottom, cup-and-handle, pennant, channel, ascending/descending triangle distinction) are priorities?
