## 3.1 — Реальный поток данных (WebSocket vs REST)

- Новый модуль `src/backend/binance_ws_feed.py` (`BinanceWebSocketFeed`):
  подписка на потоки `{symbol}@kline_{tf}`, переподключение с
  экспоненциальной задержкой (до `ws_reconnect_max_delay_sec = 30 с`),
  heartbeat (`ws_heartbeat_interval_sec = 30 с`).
- `BinanceScanner` использует WebSocket при `CONFIG.data.use_websocket = True`,
  иначе — REST-поллинг.

**Латентность WS против REST (качественно):**

| Режим | Механизм | Задержка появления свечи |
|-------|----------|--------------------------|
| REST-поллинг | Опрос каждые `poll_interval_sec = 60 с` | до ~60 с |
| WebSocket | Push-обновления по закрытию свечи | ~реальное время (доли секунды) |

WebSocket устраняет фиксированную задержку поллинга и снижает число REST-запросов
(меньше риск rate-limit). При обрыве соединения система автоматически
переподключается; при недоступности WS возможен фаллбэк на REST.

## 3.3 — Оптимизация производительности

- **Кэш структур:** `_candle_hash` хэширует весь ряд close-цен (а не несколько
  точек) — изменения в середине окна не пропускаются. Управляется флагом
  `CONFIG.data.cache_structures` (по умолчанию `True`). При неизменных свечах
  пересчёт структуры пропускается.
- **Конкурентность:** обновление свечей и пересчёт структур выполняются батчами
  по `max_concurrent_symbol_tasks = 10`. Новый `_update_all_structures_async`
  распараллеливает CPU-bound расчёт через пул потоков (numpy/scipy освобождают
  GIL); группировка по символу исключает гонки.
- **Тесты:** `tests/test_performance.py` (6 шт.) — пропуск/повтор пересчёта,
  отключение кэша, детекция изменения в середине ряда, параллельный пересчёт и
  его согласованность с последовательным.

## 3.4 — Повышение recall: cup_and_handle и triple_top

**Изменения детектора (`structure_extractor.py`):**
- *Тройная вершина:* при глубоком нэклайне (depth > 0.15 диапазона) допуск
  равенства пиков ослабляется с `0.035` до `triple_top_asymmetry_tolerance =
  0.06`; снижен штраф за асимметрию расстояний между пиками.
- *Чаша с ручкой:* добавлен `_candidate_cup_fallback` — строит чашу по
  максимумам close-цен, когда основной детектор не нашёл кандидата. Правая
  кромка = пик восстановления в окне `[0.6 .. 1.4]·span` после дна. Ручка
  опциональна (`cup_and_handle_relaxed_handle_retrace_max = 0.7`) — «чаша без
  ручки» подтверждается. Защиты: `li ≤ 0.40n`, `ri ≥ 0.55n`, расхождение
  кромок < 0.10·диапазона, симметрия ≥ 0.3.

**Результаты валидации (синтетика, переобученный ML):**

| Метрика | Результат | Цель | Статус |
|---------|-----------|------|--------|
| Каузальный recall `triple_top` | **91%** | ≥75% | ✅ |
| Каузальный recall `cup_and_handle` | **70%** | ≥65% | ✅ |
| ML FPR (на шуме, 200 образцов) | **0.00%** | ≤1% | ✅ |

Сырой (без ML) FPR ~45% — ML-фильтр полностью отсекает ложные подтверждения,
сохраняя при этом recall целевых паттернов.
